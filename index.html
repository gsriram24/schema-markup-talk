With schema markup

Structured data is how your website introduces itself to machines.

Simple analogy: imagine walking into a party with no name tag. People have to guess who you are from context — what you look like, who you're standing next to. That's what Google does with your site without schema. Now imagine wearing a name tag that says your name, your role, your company. That's schema markup — you explicitly declare what your content is, instead of hoping Google figures it out.

What is Schema Markup?

  • Shared vocabulary from Schema.org — maintained by Google, Microsoft, Yahoo, Yandex
  • Tells search engines what your content means, not just what it says
  • Three formats: JSON-LD, Microdata, RDFa — JSON-LD wins (recommended by Google)
  • Just a <script type="application/ld+json"> tag — no HTML changes needed
Schema.org is the shared vocabulary — think of it as a contract between you and search engines. JSON-LD won the format war. Why do we love it? Because it sits in a script tag in your head — completely separate from your markup. No need to sprinkle data attributes across your HTML like Microdata requires. As developers, we love separation of concerns. JSON-LD is exactly that. Drop a script tag, fill in the fields, done.

Why Should You Care?

  • Rich results — stars, FAQs, prices directly in search
  • AI search — Perplexity, ChatGPT Search, Google AI Overviews rely on structured data
  • CTR boost — not a direct ranking signal, but rich results get more clicks

Your First Schema:
Organization

{
  "@context": "https://schema.org",
  "@type": "Organization",
  "name": "Acme Corp",
  "url": "https://acme.com",
  "logo": "https://acme.com/logo.png",
  "sameAs": [
    "https://twitter.com/acme",
    "https://linkedin.com/company/acme"
  ],
  "contactPoint": {
    "@type": "ContactPoint",
    "telephone": "+1-800-555-0100",
    "contactType": "customer service"
  }
}
@context + @type are always required — every schema starts here
sameAs links your social profiles — helps build your Knowledge Graph entry
Walk through each field. @context tells the parser "we're using Schema.org vocabulary." @type says "this is an Organization." Every other field describes attributes of that entity. The sameAs field is powerful — it links your brand across platforms, helping Google build a Knowledge Graph entry for you. contactPoint is nested — it has its own @type, showing how schema represents relationships. This is the simplest schema to add. Put it on your homepage today — it takes 10 minutes.

Highest-ROI Schema: FAQPage 

{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "What is schema markup?",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "Schema markup is structured
        data you add to your website..."
    }
  }]
}
mainEntity is an array of Questions
acceptedAnswer must be an object, not a string!
This is the highest-ROI schema you can add. If you have any FAQ content on your site — and most sites do — this is free real estate in search results. Google will show expandable Q&A dropdowns right on the search page. Note the nesting carefully: mainEntity is an array, each item has @type Question, and acceptedAnswer must be an object with @type Answer — not just a string. This is one of the most common mistakes, and we'll see it on the next slides.

Spot the Bug

❌ Wrong
{
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "What is schema?",
      "acceptedAnswer": "Schema is
        structured data that..."
    }
  ]
}
✅ Correct
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "What is schema?",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "Schema is structured
          data that..."
      }
    }
  ]
}
1
Missing @context — parser doesn't know you're using Schema.org
2
acceptedAnswer is a string — must be an Answer object
3
Missing @type: "Answer" on the nested object
This is the slide where everyone gets uncomfortable. The version on the left looks totally reasonable — it's valid JSON, it has the right field names. But it fails schema validation in three ways. First: no @context. Without it, the parser doesn't know you're using Schema.org vocabulary. Second: acceptedAnswer is a plain string instead of an Answer object. Third: even if you wrap it in an object, you need @type Answer. These are the three most common schema bugs I see. They all pass a JSON linter but fail schema validation. That's the trap.

Things That Will
Bite You

  • Validation ≠ rich results — Google decides eligibility separately; passing validation is necessary but not sufficient
  • SPAs & client-side rendering — Googlebot may not execute your JS; use SSR, SSG, or prerendering for schema
  • Schema drift — content changes but schema stays stale (wrong prices, outdated FAQs, closed locations)
  • Duplicate schemas — multiple conflicting schema blocks on one page confuse parsers
  • No good all-in-one validator — I forked Adobe's structured-data-validator to support 140+ types with automatic subtype inheritance
These are things I hit firsthand while building the schema builder at HighLevel. The biggest surprise: validation passing doesn't mean Google shows rich results — they have their own eligibility criteria beyond structural correctness. The SPA issue is critical for this audience — if your React or Vue app is client-side rendered, Googlebot might not see your schema at all. Use SSR or SSG. Schema drift is insidious — someone updates a product price on the page but nobody updates the schema. Now you've got mismatched data, and Google can penalize you for that. And when I was building support for 140+ types, no single validator handled them all. Adobe's structured-data-validator was the closest, but it didn't support automatic subtype inheritance — like knowing that Restaurant inherits all the required fields from LocalBusiness. So I forked it and added that. The link is on the resources slide.

Always validate before shipping. I've seen valid JSON that was completely wrong as schema. Use two tools: the Rich Results Test tells you if Google will show rich results from your markup. The Schema.org validator checks structural correctness — does your schema follow the spec? Use both, they catch different things. After deployment, Google Search Console will show you schema errors across your site — check it regularly.
